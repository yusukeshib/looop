looop — a tiny, portable, autonomous control loop for agent-driven work.

This single binary IS the whole program: principles, rules, and defaults all
live inside it. Install one binary and run it — no README, no helper dir.

THE IDEA
  looop is an AUTONOMOUS control loop. Each beat it senses the world and, when
  something changed, decides the SINGLE most important move and executes it —
  spawning worker agents for hands-on work. The judgment lives INSIDE looop (a
  small, gated LLM call per beat); looop is the brain.

  You are a PEER, not the driver. You steer by editing goals / the PLAYBOOK
  (looop observes them next beat); a worker that hits a decision only a human can
  make asks you and waits. You reach looop through a CLIENT — anything that
  drives the contract verbs below on your behalf: a bare terminal (you, typing
  `looop …`), an AGENT client (claude/codex/opencode/pi/custom) you talk to in
  plain language, which reads state, relays asks, and edits goals for you), or a NOTIFY
  client (a script that pushes pending asks to Slack/SMS/desktop and relays your
  reply back). A client is an INTERFACE, not a decision-maker: looop decides,
  the client just talks. core knows nothing about any particular client.

    role          who                   what it does
    ----          ---                   ------------
    judgment      the pulse (looop)      sense every beat; decide + execute ONE move
    hands         worker sessions        real agents doing multi-step work; when they
                                         need a human decision they `ask` and wait
    interface     a client               OPTIONAL: drives the contract for you — read
                  (you run it)           state, relay asks, steer. Never decides.
    memory        core (the backend)     PLAYBOOK + goals + journal + mailbox state

THREE LAYERS
  core      the autonomous pulse + the durable state behind it. Decides + acts.
            State is held in a file-based backend today; that is an
            implementation detail you never address directly.
  contract  the `looop …` verbs. The ONE supported surface to read + steer
            core. This — not the backend layout — is the stable boundary.
            `up` / `down` are the porcelain a human runs by hand; the rest is
            the contract a client drives.
  client    anything that drives the contract for a human. The human reaches
            core ONLY through the contract; a bare terminal is the thinnest
            client, an agent/notify client does the same calls on your behalf.

HOW IT RUNS
  1. `looop up` — start the pulse: a detached, autonomous loop that senses,
     decides ONE move per changed beat, and runs the worker fleet. That is looop.
  2. (optional) run an agent client to watch + steer:
        claude  # or codex/opencode/pi/custom, then say: "be my looop client —
                #        read `looop --help`, show me `looop state`, relay
                #        pending asks and let me answer in plain language, help
                #        me edit goals/PLAYBOOK. looop itself decides; you never
                #        decide for it."
  3. `looop down` — stop the pulse and every live worker.
  You can skip a long-running client entirely and drive the contract by hand:
  `looop state`, `looop answer`, `looop goal write`.

ONE BEAT (sense → decide → act)
  Each beat the pulse re-runs every `sensors/*.sh` (a snapshot still fresh under
  a declared `# looop:interval=N` cadence is kept; anything a live sensor does
  not own is pruned) so `snapshots/` reflects the world. If the world is
  UNCHANGED since last beat it stops there — no LLM call, nearly free. When it
  changed, looop hands the PLAYBOOK + goals + readings + pending asks to the
  configured `tick` runner, which emits ONE typed action; looop executes it (and
  gates risky ones). It is the SOLE senser + decider (single-instance flock), so
  beats never race. Cadence is one interval (config `interval`, default 60s); a
  move may nudge the next beat sooner. Spend stays bounded even when the world
  is noisy: decide attempts are capped per rolling hour
  (LOOOP_MAX_DECIDES_PER_HOUR, default 120), the live fleet is capped
  (LOOOP_MAX_WORKERS, default 8), and a sensor whose wake signal changes every
  beat is flagged to the decider as FLAPPING so it fixes the sensor instead of
  burning a decide per beat.

============================================================================
CLIENT CONTRACT  (read this to BECOME a looop client)
============================================================================
A client never touches core directly — it drives the contract verbs below.
Everything a human does (answer asks, steer, watch) goes through these. They are
backend-agnostic: they work the same whether core's state lives in files, an
embedded DB, or anything else. A client does FOUR things:

  1. SUBSCRIBE — block cheaply until something needs a human:
       looop wait --actionable     wake on asks/journal moves
       looop wait --only-asks      wake only on a new pending ask
       looop wait [--json]         wake on any category change
     `wait` prints state + a `changed: […]` diff, so ONE call tells you WHAT
     moved — no follow-up `tail` / `jq` needed. If the pulse is DOWN it wakes
     immediately with `changed: [pulse-down]` (and state shows `pulse: DOWN`)
     instead of blocking forever on a loop that will never move the files;
     that wake EXITS 2 (a normal change exits 0), so a shell client can branch
     on the exit code without parsing stdout.
     ⚠ TWO WAKES THAT MUST NOT BE RE-ARMED BLINDLY (both return instantly,
     so a subscribe-handle-resubscribe loop becomes a tight busy-loop):
       • pulse-down — nothing will move until the pulse is restarted; tell
         the human to run `looop up`, and only resume waiting once
         `state --json` shows `pulse_alive: true` again.
       • a HELD ask — an ask the human is deliberately leaving unanswered
         keeps the pending set non-empty, so `--only-asks` returns in 0s with
         an unchanged `world_hash` every time. Stop waiting, leave the blocked
         worker parked, and check `state` on demand until the ask resolves.

  2. READ — inspect without blocking:
       looop state [--json]        full world: goals, snapshots, fleet, asks
       looop asks  [--json]        JUST pending asks (id, prompt, ref, options)

  3. ANSWER — resolve a worker blocked on a human decision:
       looop answer <ask_id> "<text>"     inline
       looop answer <ask_id> -            multi-line from stdin / heredoc
     Each ask carries an id, the prompt, an optional `ref` (an artifact to read
     first) and optional `options` (discrete choices — map these to buttons /
     numbered replies; free-text answers are always allowed). IRREVERSIBLE asks
     (merge / deploy / delete) MUST get an explicit human yes — never
     auto-answer them.
     ⚠ A worker blocked on an ask is NOT idle — never kill it; answer it or
     leave it parked (killing strands the eventual answer and loses context).
     A DETACHED ask (the worker checkpointed its state and exited by design)
     needs nothing special from you: answer it like any other; looop's decider
     then re-dispatches a fresh worker with your answer and the checkpoint
     (`worker start --resume <ask_id>`). Answering a NON-detached ask whose
     worker already EXITED only records the answer — the dead worker won't act
     on it; spawn a fresh worker instead.

  4. STEER — shape WHAT looop pursues (takes effect next beat):
       looop goal write <id> [body|-]   |   looop goal archive <id>
       looop playbook write [body|-]
       looop sensor write <name> [script|-]
       looop schedule write <name> --in S | --every S [--note …]   durable time
         trigger (one-shot / recurring, min 60s) that WAKES the loop when due;
         looop schedule rm <name> | schedule list [--json]
       looop tell <worker> "<msg>"      queue steering INTO a live worker — it
         picks the message up at its next `told` check or with its next answer
     These are the supported steering surface: validated, journaled, and
     single-writer-safe. (The backend also keeps goals/PLAYBOOK/sensors as plain
     files you can git + inspect; editing those files directly works and is seen
     next beat, but skips the journal entry — it is an escape hatch, not the
     interface. Clients steer through the verbs.)

  Optional peek verbs for the fleet (read-only):
       looop worker list [--all] [--watch]   fleet + health, idle/uptime/ask age
       looop screenshot <id>                 capture what a worker shows right now
     health: busy (producing output) · waiting-ask (the human's turn) · stuck
     (alive, no ask, silent ≥ LOOOP_WORKER_STUCK_SECS — the pulse kills these)
     · dead. A worker's ONLY input channel is the ask/answer mailbox — there is
     deliberately no way to type into its terminal, so every human decision
     leaves a durable ask/answer record.

  BUILD A NOTIFY CLIENT — no core changes; just drive the contract in a loop:
       while looop wait --only-asks --json >/tmp/ev; do
         # push each new ask to your channel — CARRY THE ASK ID so the reply
         # correlates — then, on the human's reply:
         looop answer "<ask_id>" "<reply>"
       done
     Push once per ask; escalate on age. Route by urgency: desktop/digest for
     low, chat for normal, SMS/voice for irreversible.

  BUILD AN AGENT CLIENT (a "concierge") — any agent session pointed at looop:
  have it `wait`, relay pending asks to you in plain language, `answer` your
  reply, and edit goals/PLAYBOOK via the write verbs. It interfaces; looop decides.

WORKER CONTRACT (auto-injected into every worker — for reference)
  A worker that needs a QUICK human decision runs ONE blocking call and waits:
     answer=$("$LOOOP_BIN" ask "$LOOOP_SESSION_ID" --prompt "…" [--ref P] [--options a,b])
  It needs no terminal/attach — `ask` returns when answered (a client replies via
  `looop answer`). For a LONG wait (hours+), the worker instead CHECKPOINTS its
  state to reports/, raises the ask with `--detach` (returns the ask id, no
  blocking), and ENDS its session — the answer is delivered to a FRESH worker
  the decider dispatches with `--resume <ask_id>` (question + answer +
  checkpoint injected into its brief; the pair is archived). Workers write only
  to claims/, reports/ and their own code sandbox; they end themselves with
  `kill`, and lease shared resources with `claim`/`unclaim`.

CODE / CONFIG / DATA are cleanly separated (all overridable by env)
  EXEC    this one binary                          the program (portable)
  CONFIG  <DATA>/config.json                       one file: runner wiring
          └ override with $LOOOP_CONFIG            (seeded inline if absent)
            A runner needs TWO commands — `tick_command` (run one disposable
            decide move, prompt on stdin) and `worker_command` (launch a worker
            session; `{{prompt_file}}` is its brief). Optional: `interval`
            (cadence), `session_ttl` (corpse retention).
  DATA    ${XDG_STATE_HOME:-~/.local/state}/looop/  the file-based backend
          └ override with $LOOOP_DATA_DIR
            PLAYBOOK.md  goals/  journal.md  sensors/*.sh   <- durable memory
            asks/ answers/   <- the worker <-> human mailbox (surfaced by clients)
            verify/          <- worker post-conditions (<id>.cmd) + verdicts
                                (<id>.json): a shell command declared at spawn,
                                run ONCE by the pulse after the worker dies —
                                exit status alone is never trusted as "done"
            playbook.d/      <- PLAYBOOK history: the previous body is
                                snapshotted here before every overwrite
                                (LOOOP_PLAYBOOK_KEEP generations, default 20)
            snapshots/ runs/ .lock .last-tick-hash .tick-backoff  <- scratch
            reports/         <- worker deliverables a human reads (persist)
            sessions/<id>/   <- worker + pulse sessions (babysit state, per
                                profile; auto-reaped after session_ttl, 3d).
          This layout is the current backend, reached THROUGH the contract — not
          a public interface. Steer with the verbs, not by writing these files.

BOOTSTRAP
  Run `looop init` once: choose claude, codex, opencode, pi, or custom wiring
  (tick_command/worker_command). `looop up` requires this. A fresh
  data dir is also seeded once with a starter PLAYBOOK + goals/setup.md +
  goals/playbook-daily.md + sensors/today.sh + a real pending setup ask, all
  embedded in this binary. The ask wakes a client/concierge waiting on asks so it
  can interview you, then rewrite the seed into your real goals + PLAYBOOK.

DEPENDENCIES
  Just the configured agent CLI (claude/codex/opencode/pi/custom, chosen via
  `looop init`; claude by default) — used BOTH for looop's per-beat decide (`tick`) and
  to launch workers (and, if you run one, an agent client).
  Session management (babysit) is LINKED IN as a library and runs in-process; no
  babysit binary is required. A worker that touches code makes its own sandbox
  (box if available, else git worktree).

ENVIRONMENT KNOBS (all optional; unset = the default shown; an unparseable
  value warns and falls back to the default)
  wiring
    LOOOP_DATA_DIR              data dir override (profiles)   default: XDG state
    LOOOP_CONFIG                config file override           default: <DATA>/config.json
    LOOOP_LOG_FORMAT=json       structured JSON event stream   default: human lines
    LOOOP_BIN / LOOOP_SESSION_ID / LOOOP_CLAIMS_DIR / LOOOP_REPORTS_DIR
                                exported BY looop for its children (sensors,
                                workers, the runner pipeline) — read, don't set
  cadence & AI spend
    LOOOP_INTERVAL              seconds between beats          default: 60
                                                               (or config `interval`)
    LOOOP_MAX_NEXT_INTERVAL     cap on the decider's one-shot  default: 300
                                next_interval_s nudge (secs);  (or config
                                the pulse is blind to world     `max_next_interval`)
                                changes during an AI nap, so
                                this bounds that window
    LOOOP_MAX_DECIDES_PER_HOUR  decide-attempt ceiling         default: 120 (0 = off)
    LOOOP_NOOP_TTL              re-decide an aged noop (secs)  default: 21600 (0 = off)
    LOOOP_FLAP_STREAK           beats before "flapping" flag   default: 5 (0 = off)
  timeouts & limits
    LOOOP_TICK_TIMEOUT_SECS     decide-runner deadline         default: 1800 (0 = off)
    LOOOP_SHELL_TIMEOUT_SECS    run_shell action deadline      default: 300
    LOOOP_SHELL_ALLOW_DANGEROUS=1  bypass run_shell deny-list  default: off
    LOOOP_SENSOR_TIMEOUT        per-sensor deadline (secs)     default: 60
    LOOOP_SENSOR_MAX_BYTES      per-snapshot size cap          default: 8192
    LOOOP_VERIFY_TIMEOUT_SECS   per-verify deadline            default: 60
    LOOOP_VERIFY_BEAT_BUDGET_SECS  all verifies, one beat      default: 120
    LOOOP_MAX_WORKERS           standing live-fleet cap        default: 8 (0 = off)
    LOOOP_WORKER_STUCK_SECS     silent-worker threshold        default: 900
  retention
    LOOOP_RUNS_KEEP             runs/ replay dirs kept         default: 50 (0 = all)
    LOOOP_PLAYBOOK_KEEP         playbook.d/ generations        default: 20 (0 = all)
    LOOOP_SESSION_TTL           session corpse TTL (secs)      default: 259200
                                                               (or config `session_ttl`)
    LOOOP_EVENTS_MAX_BYTES      events.jsonl rotate size       default: 5242880 (0 = off)
    LOOOP_JOURNAL_MAX_BYTES     journal.md rotate size         default: 5242880 (0 = off)
  client polling
    LOOOP_WAIT_POLL_MS          `looop wait` poll interval     default: 1000
    LOOOP_ASK_POLL_MS           blocking-ask poll interval     default: 1000
    LOOOP_ASK_TIMEOUT_S         blocking-ask wall clock        default: none (waits forever)
  deps preflight
    LOOOP_TICK_BIN / LOOOP_WORKER_BIN  conventional names for env vars YOU may
                                reference from config.json commands (e.g.
                                tick_command: '"$LOOOP_TICK_BIN" -p') so one
                                wiring can point tick/worker at different
                                binaries. looop sets neither — your shell/env
                                does; the shell expands them at run time, and
                                the deps preflight skips (never hard-gates on)
                                commands invoked through a variable.
